EXECUTIVE SUMMARY
1.1 Project Overview
BrainBox is a focused, AI-enhanced learning workspace designed for understanding and long-term retention. Unlike general-purpose document editors, BrainBox is a structured learning engine where every feature exists to reduce distraction and strengthen comprehension. Content is organized into Notebooks, providing the structure required for AI assistance, quiz generation, and text-to-speech review.
1.2 Objectives
Build a notebook editor that supports rich text content managed by an outline generated from headers.
Implement a focused editor for deep work and comprehension.
Integrate a context-aware AI assistant capable of modifying notebook content, generating summaries, quizzes, and brainstorming prompts.
Develop a read-only Review Mode with built-in text-to-speech and playback controls.
Deliver consistent experiences across a React web application and a native Android mobile app, backed by a Spring Boot API
1.3 Scope
Included Features:
User authentication with JWT (register, login, logout)
Library for organizing notebooks by category
Rich text document editor with header-based outline navigation.
AI assistant for content expansion, summaries, explanations, quiz generation, and structural improvements.
Manual and AI-generated quiz creation
Read-only Review Mode with text-to-speech and playback controls.
Responsive web interface built with React
Native Android mobile application built with Kotlin and Jetpack Compose
PostgreSQL database with relational schema
Version control for notebook history and revisions.
Excluded Features:
Real-time collaborative editing
Third-party cloud storage import (Google Drive, Notion, etc.)
Flashcard spaced repetition scheduling
Gamification or achievement systems
Social sharing or public notebook publishing
Email notification system
Advanced analytics dashboard

1.0 INTRODUCTION
1.1 Purpose
This document serves as the comprehensive design specification for the BrainBox system. It provides detailed requirements, architectural decisions, API contracts, database design, and implementation roadmap to guide development and ensure all components: backend, web frontend, and mobile, integrate seamlessly into a cohesive learning platform.

2.0 FUNCTIONAL REQUIREMENTS SPECIFICATION
2.1 Project Overview
Project Name: BrainBox
Domain: EdTech / Personal Knowledge Management
Primary Users: 
Students, self-directed learners.
Professionals studying reference material.
Problem Statement: Learners struggle to retain information from long, unstructured documents. Existing editors prioritize creation over comprehension, offering no meaningful integration between note organization, review, and active recall tools.
Solution: A focused notebook environment where content is supported by an AI assistant and a dedicated Review Mode — so the platform actively helps users understand and remember what they write.
2.2 Core User Journeys
Journey 1: Notebook Creation & Structured Editing
The user visits the web application and creates an account
From the Dashboard or Library, the user creates a new Notebook
The user assigns a title and selects a Category
The Notebook Editor opens, allowing the user to begin writing
The user adds content using the rich text toolbar for formatting and structure
The user uses headers to organize content, which automatically populates a navigable outline
The user uses the outline to jump to different parts of the notebook
The user continues expanding the notebook by adding and refining content
The AI Assistant is used to expand content, improve structure, or generate ideas directly within the document
Journey 2: Active Study Session
A logged-in user opens an existing Notebook from the Library
Activates Review Mode for a read-only pass through the material
Uses the outline in Review Mode to navigate between sections of content
Uses the AI assistant to request a summary of the notebook
Generates an AI quiz to test recall on the most difficult topics
Reviews quiz results and jumps back to relevant content from the results view
Journey 3: Audio Review on the Go
User opens the Review Mode on the app
Enables text-to-speech playback for the notebook
Uses the outline to jump to specific parts of the audio review
Pauses and resumes playback as needed; the system remembers the last played position
Journey 4: Quiz & Flashcard Practice
User navigates to the Quiz or Flashcards tab of a Notebook
Requests an AI-generated deck scoped to the notebook
Takes the quiz or reviews flashcards
Saves the session progress to track mastery over time
Journey 5: Library Management
The user opens the Library to organize their collection
Uses Search and Category filters to find specific Notebooks
Assigns Notebooks to relevant Categories
Moves or renames Notebooks to maintain organization
Deletes old or unneeded Notebooks to keep the workspace clean
Journey 6: Study Playlists & Continuous Playback
User navigates to the Playlists section to prepare for a long study session
Creates a new Study Playlist for a specific subject or exam
Adds multiple Notebooks to the Playlist for continuous listening
Reorders the Playlist to prioritize specific topics
Starts the Playlist to begin a hands-free audio review session

2.4 Detailed Feature Specifications
Feature: User Authentication
Screens: Registration, Login, Forgot Password
Fields: Username, Email, Password, Confirm Password
Validation: Email format, password strength, uniqueness
API Endpoints: POST /api/auth/register, POST /api/auth/login, POST /api/auth/logout, GET /api/auth/verify-email, POST /api/auth/forgot-password, POST /api/auth/verify-code, POST /api/auth/reset-password
Security: Stateless JWT authentication with access and refresh tokens. Passwords hashed with bcrypt. Email verification via Resend.
Feature: Dashboard
Screens: Dashboard (home screen).
Functions: Welcome greeting. "Continue learning" area for the two most recently edited notebooks. "Recently Reviewed" area for quick access to audio review. Quick actions for notebook creation.
API Endpoints: GET /api/notebooks
Feature: Notebook Management
Screens: Library
Functions: Primary hub for notebook management. Create, rename, delete notebooks. Organize notebooks into categories. Search, filter, and sort (A-Z, Last Edited, Last Reviewed). 
API Endpoints: GET /api/notebooks, POST /api/notebooks, PUT /api/notebooks/{id}, DELETE /api/notebooks/{id}, GET /api/categories, POST /api/categories.
Feature: Editor
Screens: Notebook editor.
Structure: Each Notebook contains rich text content organized by headers.
Outline: Dynamic navigation outline generated automatically from document headers (H1, H2, H3). Clicking an outline item scrolls the editor to the corresponding header.
Editor Formatting: Bold, italic, bullet lists, numbered lists, quote blocks, code blocks, basic alignment. No decorative fonts, text colors, or complex layout tools.
Feature: Version Control
Functions: Automatically save revisions of the notebook content. View history of changes. Restore previous versions.
API Endpoints: GET /api/notebooks/{id}/versions, POST /api/notebooks/{id}/versions, POST /api/notebooks/{id}/versions/{versionId}/restore.
Feature: AI Assistant
Screens: AI panel (slide-in or docked panel within the editor).
Context Scope: Full Notebook.
Capabilities: Modify and expand document content, explain unclear sections, simplify complex paragraphs, generate short/medium/detailed summaries, suggest structural improvements, brainstorm related ideas.
Conversation History: Each notebook maintains its own AI conversation thread stored per user.
API Endpoints: POST /api/ai/chat, GET /api/ai/history/notebook/{id}, DELETE /api/ai/history/notebook/{id}.
Feature: Quiz
Screens: Quiz builder, quiz-taking view, results view.
Creation Modes: Manual (user writes questions and answer options) or AI-generated (scoped to the full notebook).
Question Types: Multiple choice, true/false (initial version).
Functions: Edit, reorder, delete individual questions. Save quiz to notebook. Take quiz and receive a score. Review correct and incorrect answers.
API Endpoints: GET /api/quizzes/notebook/{id}, POST /api/quizzes, PUT /api/quizzes/{id}, DELETE /api/quizzes/{id}, POST /api/ai/generate-quiz, POST /api/quizzes/{id}/attempts.
Feature: Study Playlists
Screens: Playlists screen, Playlist builder, Application-wide (PlayerBar).
Functions: Create, edit, and delete study playlists. Add notebooks to playlists for multi-notebook audio review sessions. Reorder items within a playlist. Global play/pause and skip controls that persist across navigation.
API Endpoints: GET /api/playlists, POST /api/playlists, PUT /api/playlists/{id}, DELETE /api/playlists/{id}, POST /api/playlists/{id}/add-notebook.
Feature: Review Mode
Screens: Read-only review screen (full screen, no editor chrome).
Functions: Focused read-only rendering of Notebook content. Dynamic outline navigation based on document headers. Text-to-speech playback of the full document. Play, pause, resume, adjust speed. Highlighted text follows audio during playback. Automatically saves the user's current playback position (timestamp/progress).
API Endpoints: POST /api/notebooks/save-progress/{id}, PATCH /api/notebooks/update-review/{id} TTS is client-side using the Web Speech API (web) or Android TTS engine (mobile).
2.5 Acceptance Criteria
AC-1: Notebook Creation and Versioning
Given I am a logged-in user, 
When I create a Notebook 
And add content with headers, 
Then the outline should reflect the document structure,
And I should be able to view and restore previous versions of the content.
AC-2: AI Content Modification
Given I am in the editor, 
When I ask the AI to expand or improve a section of the document, 
Then the AI should be able to update the document content directly.
AC-3: AI Summary
Given I am in the editor, 
When I request a summary, 
Then the AI should return a summary based on the notebook content.
AC-4: AI-Generated Quiz
Given I have a Notebook with content,
When I request an AI-generated quiz, 
Then BrainBox should return a set of relevant questions with answer options that I can edit, save, and take.
AC-5: Review Mode with TTS and Outline
Given I activate Review Mode on a Notebook, 
When I press Play, then the system should read the document aloud while highlighting the corresponding text, 
And I can navigate to any header using the outline without exiting Review Mode,
And I can jump directly to a specific timestamp using the playback timeline,
And playback resumes from the selected point.
And the system saves my last playback timestamp so that when I return to Review Mode, playback continues from where I left off.
AC-6: Version Restoration
Given I have multiple versions of a Notebook,
When I select a previous version from the history,
Then the editor should preview that version,
And when I click Restore, the current content should be replaced by that version.
AC-7: Study Playlists
Given I have multiple notebooks in my library,
When I create a new Study Playlist,
And add multiple notebooks to it,
Then I should be able to start continuous playback of the playlist items,
And reorder the items to change the playback sequence.
3.0 NON-FUNCTIONAL REQUIREMENTS
3.1 Performance Requirements
API response time must be at or under 2 seconds for 95% of requests. 
Web page and view transitions must complete within 2 seconds. 
AI response streaming should begin within 3 seconds of the request. 
Mobile app cold start must complete within 3 seconds. 
The system must support 100 concurrent users. Database queries must complete within 500ms.
3.2 Security Requirements
HTTPS for all client-server communication. 
Stateless JWT authentication using short-lived access tokens and longer-lived refresh tokens. 
Refresh token rotation on every use. 
Passwords hashed using bcrypt with a minimum of 12 salt rounds. 
SQL injection prevention through parameterized queries and JPA. 
XSS protection via output encoding and Content Security Policy headers. R
Rate limiting at 100 requests per minute per authenticated user and 30 requests per minute per IP for unauthenticated endpoints. 
AI endpoints must validate that the requesting user owns the notebook being referenced.
3.3 Compatibility Requirements
Web Browsers: Chrome, Firefox, Safari, Edge (latest 2 versions)
Android: API Level 24+ (Android 7.0+)
Screen Sizes: Mobile (360px+), Tablet (768px+), Desktop (1024px+)
Operating Systems: Windows 10+, macOS 10.15+, Linux Ubuntu 20.04+
3.4 Usability Requirements
A first-time user should be able to create a notebook, add content with headers, and see an auto-generated outline within 5 minutes. 
WCAG 2.1 Level AA compliance for the web application. 
Consistent navigation and information architecture across web and mobile.
Clear, actionable error messages with recovery guidance. 
Touch targets on mobile must be a minimum of 44×44px. 
Review Mode must be fully keyboard-navigable on web.

6.0 DATABASE DESIGN
6.1 Entity Relationship Diagram

Detailed Relationships:
One-to-Many: users → notebooks (a user owns multiple notebooks).
One-to-Many: users → categories (a user organizes notebooks into multiple categories).
One-to-Many: notebooks → versions (a notebook has history of content).
One-to-Many: users → refresh_tokens (handles multi-device session management).
One-to-Many: notebooks → quizzes (notebooks serve as the primary container for quizzes).
One-to-Many: notebooks → flashcard_decks (notebooks serve as the primary container for decks).
One-to-Many: quizzes → quiz_questions → quiz_options (structured quiz content).
One-to-Many: users × quizzes → quiz_attempts (tracks user performance over time).
One-to-Many: notebooks → flashcard_decks → flashcards (organized study cards for active recall).
One-to-Many: notebooks → ai_history_entries (persists AI chat context per notebook).
One-to-Many: users → study_playlists (users can create multiple saved playlists).
One-to-Many: study_playlists → playlist_items (manages notebooks within a playlist).
Key Tables:
users: User accounts and authentication credentials.
refresh_tokens: JWT refresh token storage with rotation support and device tracking.
categories: User-defined notebook categories for library organization.
notebooks: Core notebook records owned by users. Includes last_reviewed_at timestamp.
notebook_versions: Saved versions of notebook content.
study_playlists: Collections of notebooks saved for continuous audio review.
playlist_items: Ordered list of notebooks within a specific study playlist.
quizzes: Quiz sets linked to a notebook.
quiz_questions: Individual questions belonging to a quiz.
quiz_options: Answer options for each question, with correct answer flag.
quiz_attempts: Records of a user taking a quiz, including score.
quiz_answers: Per-question answer selections within a quiz attempt.
flashcard_decks: Collections of study cards linked to a notebook.
flashcards: Individual study cards with front and back text for active recall.
ai_history_entries: Per-notebook AI conversation history scoped to a user and mode.
Table Structure Summary:
users: id (PK), username, email, password, role, banned, verified, last_login, last_logout, created_at.
refresh_tokens: id (PK), user_id (FK), token, expiry_date, user_agent, ip_address, created_at.
categories: id (PK), user_id (FK), name, created_at, updated_at.
notebooks: id (PK), user_id (FK), category_id (FK, nullable), title, content (TEXT), created_at, updated_at, last_reviewed_at.
notebook_versions: id (PK), notebook_id (FK), content (TEXT), created_at.
study_playlists: id (PK), user_id (FK), name, created_at, updated_at.
playlist_items: id (PK), playlist_id (FK), notebook_id (FK), order_index, created_at.
quizzes: id (PK), notebook_id (FK), title, created_by_ai (boolean), created_at.
quiz_questions: id (PK), quiz_id (FK), question_text, question_type (multiple_choice, true_false), order_index.
quiz_options: id (PK), question_id (FK), option_text, is_correct (boolean).
quiz_attempts: id (PK), quiz_id (FK), user_id (FK), score, total, attempted_at.
quiz_answers: id (PK), attempt_id (FK), question_id (FK), selected_option_id (FK), is_correct (boolean).
flashcard_decks: id (PK), notebook_id (FK), title, created_at.
flashcards: id (PK), deck_id (FK), front_text, back_text, difficulty_score, last_reviewed_at, created_at.
ai_history_entries: id (PK), notebook_id (FK), user_id (FK), role (user/assistant), content (TEXT), mode (summarize/explain/brainstorm), created_at.
