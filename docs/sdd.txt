EXECUTIVE SUMMARY
1.1 Project Overview
BrainBox is a study workspace designed around the full learning loop: capture ideas in structured notebooks, review them in a focused read-only mode, reinforce understanding through quizzes and flashcards, and continue listening through playlist-based audio review. The platform combines notebook authoring, library organization, study tools, and per-notebook AI assistance into one product model shared across the backend, web application, and mobile experience. 
1.2 Objectives
Provide a structured notebook editor with rich formatting, outline navigation, and recoverable version history. 
Support a focused review experience with text-to-speech, synced highlighting, playback controls, and resume behavior.
Enable quiz and flashcard study flows tied to notebook content, with attempt tracking for repeated practice. 
Keep notebooks organized through categories, search, sorting, and named playlists for continuous listening. 
Offer secure account management, profile settings, and per-user AI configuration for OpenAI-compatible providers. 
Spring Boot REST API consumed by a React web app and a Kotlin Android app.

1.3 Scope
Included Features:
User authentication with registration, login, logout, email verification, password reset, refresh token support, and Google sign-in. 
Notebook creation, editing, categorization, recent-study surfaces, version snapshots, preview, restore, and export. 
Library organization through category management, notebook search, sorting, batch actions, and notebook reassignment. 
Review Mode with outline navigation, device text-to-speech, synced highlighting, playback speed control, and saved listening progress. 
Quizzes: CRUD with multiple-choice questions, attempt scoring.
Quiz and flashcard creation, editing, study, and attempt tracking, including AI assisted generation within the notebook study flow. 
Named playlists with queue management, reordering, and current-item tracking for continuous listening sessions. 
AI conversations and AI configuration management with one selectable active provider profile per user.
Profile management across web and mobile, including username/email updates and password changes. 
Mobile integration that extends authentication, dashboard continuation, review, playlists, and study access. 
Excluded Features:
Real-time collaborative editing
Third-party cloud storage import (Google Drive, Notion)
Spaced-repetition scheduling
Social sharing / public notebooks
Advanced analytics

1.0 INTRODUCTION
1.1 Purpose
This document serves as the primary design specification for BrainBox. It defines the system requirements, architecture, API contracts, and database schema to guide development of the backend, web frontend, and mobile application. All components — Spring Boot API, React web app, and Kotlin Android app — are expected to be built in accordance with this specification.

2.0 FUNCTIONAL REQUIREMENTS SPECIFICATION
2.1 Project Overview
Project Name: BrainBox
Domain: EdTech / Personal Knowledge Management
Primary Users: 
Students, self-directed learners.
Professionals studying reference material.
Problem Statement: earners often write notes in one tool, review them in another, and practice recall somewhere else. That fragmentation weakens continuity, makes review inconsistent, and turns study materials into static documents instead of reusable learning assets. 
Solution: BrainBox treats each notebook as a study asset that can be written, organized, reviewed, listened to, discussed with AI, and transformed into quizzes, flashcards, and playlists without leaving the platform. 
2.2 Core User Journeys
Journey 1: Notebook Creation & Structured Editing
The user signs in and starts a notebook from the Dashboard or Library. 
The user assigns a title, selects a category, and opens the notebook editor.
The user structures content with headings, lists, tables, links, highlights, and other supported rich-text tools. 
The outline updates automatically from the document headings and is used for quick navigation while editing.
The AI assistant helps refine, expand, or reorganize content inside the notebook when the user requests it. 
The user saves progress, creates manual snapshots when needed, and can preview or restore previous versions later. 
Journey 2: Review Mode & Guided Listening 
The user opens an existing notebook and switches to Review Mode. 
BrainBox renders the notebook in a read-only study layout with the outline available on the side. 
The user starts text-to-speech playback, adjusts speed, and moves between sections using the outline. 
Highlighted content follows the active playback position while the player preserves the user's place for later continuation. 
The AI assistant remains available in a read-only study mode for explanations, summaries, and study-asset generation without directly rewriting notebook content. 
Journey 3: Quiz & Flashcard Practice 
The user navigates from a notebook to its associated quiz or flashcard deck, or creates a new deck linked to that notebook. 
The study set may be authored manually or generated from notebook context through the AI assistant. 
For a quiz, the user answers each question and submits the attempt to receive a score; for a flashcard deck, the user flips through cards and marks their mastery level at the end. 
The user reviews their score or mastery result and can view attempt history to track improvement over time. 
The user can return to the source notebook to edit content and regenerate study materials through the AI assistant to continue the learning cycle. 
Journey 4: Library Organization & Playlist-Based Listening 
The user opens the Library to search, sort, and browse notebooks across categories, using filters and sort options to quickly locate specific notebooks. 
The user assigns or changes the category of a notebook, and can batch-select multiple notebooks to reorganize, move, or delete them at once. 
The user creates a named playlist and adds one or more notebooks to build a study queue. 
The user reorders notebooks in the playlist to match the intended study sequence. 
The user starts a playlist session: notebooks play in sequence through review mode, and the current position within the playlist is preserved so the session can be resumed exactly where it left off.

2.5 Acceptance Criteria
AC-1: Notebook Creation and Versioning
Given I am a logged-in user, 
When I create a Notebook 
And add content with headers, 
Then the outline should reflect the document structure,
And I should be able to view and restore previous versions of the content.
AC-2: AI Content Modification
Given I am in the editor, 
When I ask the AI to expand or improve a section of the document, 
Then the AI should be able to update the document content directly.
AC-3: AI Summary
Given I am in the editor, 
When I request a summary, 
Then the AI should return a summary based on the notebook content.
AC-4: AI-Generated Quiz
Given I have a Notebook with content,
When I request an AI-generated quiz, 
Then BrainBox should return a set of relevant questions with answer options that I can edit, save, and take.
AC-5: Review Mode with TTS and Outline
Given I activate Review Mode on a Notebook, 
When I press Play, then the system should read the document aloud while highlighting the corresponding text, 
And I can navigate to any header using the outline without exiting Review Mode,
And I can jump directly to a specific timestamp using the playback timeline,
And playback resumes from the selected point.
And the system saves my last playback timestamp so that when I return to Review Mode, playback continues from where I left off.
AC-6: Version Restoration
Given I have multiple versions of a Notebook,
When I select a previous version from the history,
Then the editor should preview that version,
And when I click Restore, the current content should be replaced by that version.
3.0 NON-FUNCTIONAL REQUIREMENTS
3.1 Performance Requirements
API response time must be at or under 2 seconds for 95% of requests. 
Web page and view transitions must complete within 2 seconds. 
AI-assisted actions should return within a study-appropriate time budget and surface clear loading states while responses are being prepared. 
Mobile app cold start must complete within 3 seconds. 
The system must support 100 concurrent users. Database queries must complete within 500ms. 
3.2 Security Requirements
HTTPS for all client-server communication. 
Stateless JWT authentication using short-lived access tokens and longer-lived refresh tokens. 
Refresh tokens must be validated and renewed through the dedicated authentication flow. 
Passwords hashed using bcrypt with a minimum of 12 salt rounds. 
SQL injection prevention through parameterized queries and JPA. ● User input must be validated and safely rendered to reduce XSS and malformed-content risks. 
Sensitive AI provider credentials must be stored and exposed in a masked form appropriate for authenticated settings management. 
AI endpoints must validate that the requesting user owns the notebook being referenced. 
3.3 Compatibility Requirements
Web Browsers: Chrome, Firefox, Safari, Edge (latest 2 versions) 
Android: API Level 24+ (Android 7.0+) 
Screen Sizes: Mobile (360px+), Tablet (768px+), Desktop (1024px+) 
Operating Systems: Windows 10+, macOS 10.15+, Linux Ubuntu 20.04+
3.4 Usability Requirements 
A first-time user should be able to create a notebook, add content with headers, and see an auto-generated outline within 5 minutes. 
WCAG 2.1 Level AA compliance for the web application. 
Consistent navigation and information architecture across web and mobile.
Clear, actionable error messages with recovery guidance. 
Touch targets on mobile must be a minimum of 44×44px. 
Review Mode must be fully keyboard-navigable on web.

5.3 Error Handling
HTTP Status Codes
200 OK - Successful request
201 Created - Resource created
400 Bad Request - Invalid input
401 Unauthorized - Authentication required/failed
403 Forbidden - Insufficient permissions
404 Not Found - Resource doesn't exist
409 Conflict - Duplicate resource
500 Internal Server Error - Server error

Error Code Examples

Example 1
{
  "success": false,
  "data": null,
  "error": {
    "code": "BAD_REQUEST",
    "message": "Invalid credentials",
    "details": null
  },
  "timestamp": "2024-01-28T10:30:00Z"
}
Example 2
{
  "success": false,
  "data": null,
  "error": {
    "code": "NOT_FOUND",
    "message": "Resource not found", 
    "details": null
  },
  "timestamp": "2024-01-28T10:30:00Z"
}



Common Error Codes
BAD_REQUEST: Returned for IllegalArgumentException and IllegalStateException (HTTP 400)
NOT_FOUND: Returned for NoSuchElementException (HTTP 404)
FORBIDDEN: Returned for ForbiddenException (HTTP 403)
INTERNAL_SERVER_ERROR: Returned for unhandled RuntimeException and Exception (HTTP 500)
401 Unauthorized: Missing or invalid access token; handled by the security layer (not via exception handler)

6.0 DATABASE DESIGN
6.1 Entity Relationship Diagram

Detailed Relationships:
One-to-Many: users → notebooks (a user owns multiple notebooks).
One-to-Many: users → categories (a user organizes notebooks into categories). 
One-to-Many: notebooks → notebook_versions (snapshot history per notebook). 
One-to-Many: users → refresh_tokens (multi-device session management). 
One-to-One: users ↔ codes (password reset code per user). 
One-to-Many: users → quizzes; notebooks → quizzes (optional notebook link). 
One-to-Many: quizzes → quiz_questions (ordered questions with answer options stored as an element collection). 
One-to-Many: users × quizzes → quiz_attempts (tracks user performance over time). 
One-to-Many: users → flashcards; notebooks → flashcards (optional notebook link). Flashcards → flashcard_cards (ordered front/back pairs). 
One-to-Many: users × flashcards → flashcard_attempts (mastery tracking per deck). 
One-to-Many: users → playlists. Many-to-Many: playlists ↔ notebooks (via playlist_notebooks join table, ordered). 
One-to-Many: users → ai_configs (multiple provider configs per user; one selected as active). 
One-to-Many: users → ai_conversations (per-notebook, per-mode conversation history). 
Key Tables:
users: User accounts, authentication credentials, and active AI config reference. 
refresh_tokens: JWT refresh token storage with expiry and device tracking. 
codes: Hashed 6-digit password reset codes with 5-minute expiry. 
categories: User-defined notebook categories for library organization. 
notebooks: Core content records. Tracks version counter, last reviewed timestamp, and UUID for stable external references. 
notebook_versions: Immutable content snapshots keyed by timestamp. 
flashcards: Flashcard decks with ordered cards. Optional link to a notebook. 
flashcard_attempts: Mastery score (0–100) recorded per user per deck. 
quizzes: Quiz sets with difficulty level. Optional link to a notebook. 
quiz_questions: Multiple-choice questions with options stored as an ordered element collection and a correctIndex field. 
quiz_attempts: Score records per user per quiz. 
playlists: Named queues of notebooks. Stores currentIndex to track review progress. 
ai_configs: Per-user AI provider configurations (model, proxyUrl, AES encrypted apiKey). 
ai_conversations: Per-notebook, per-mode conversation history stored as a serialized JSON message array. 
Table Structure Summary:
users: id (PK), uuid, username, email, password (BCrypt), auth_provider, google_id, role, banned, verified, last_login, last_logout, created_at, selected_ai_config_id (FK, nullable). 
refresh_tokens: id (PK), user_id (FK), token (unique), expiry_date, user_agent, ip_address, created_at. 
codes: id (PK), user_id (FK, unique), code (hashed), expiry_date, created_at. 
categories: id (PK), user_id (FK), name, created_at, updated_at. 
notebooks: id (PK), uuid (unique), user_id (FK), category_id (FK, nullable), title, content (TEXT), version, created_at, updated_at, last_reviewed_at.
notebook_versions: id (PK), notebook_id (FK), content (TEXT), version (timestamp). 
flashcards: id (PK), uuid (unique), user_id (FK), notebook_id (FK, nullable), title, description, created_at, updated_at. 
flashcard_cards: id (PK), flashcard_id (FK), front (TEXT), back (TEXT), position. 
flashcard_attempts: id (PK), flashcard_id (FK), user_id (FK), mastery (int 0– 100), created_at. 
quizzes: id (PK), uuid (unique), user_id (FK), notebook_id (FK, nullable), title, description, difficulty, created_at, updated_at. 
quiz_questions: id (PK), quiz_id (FK), type, text (TEXT), options (element collection, ordered), correct_index. 
quiz_attempts: id (PK), quiz_id (FK), user_id (FK), score, created_at. 
playlists: id (PK), uuid (unique), user_id (FK), title, current_index, created_at, updated_at. 
playlist_notebooks: playlist_id (FK), notebook_id (FK), position (order column). 
ai_configs: id (PK), user_id (FK), name, model, proxy_url, api_key (TEXT, AES encrypted), created_at, updated_at. 
ai_conversations: id (PK), uuid (unique), user_id (FK), notebook_uuid, mode, title, messages (TEXT, JSON array), created_at, updated_at. 



7.0 UI/UX DESIGN
7.1 Web Application Wireframes


Dashboard (Home Screen) 
Primary layout combines a persistent sidebar, top-level workspace navigation, and study-focused dashboard cards. The main content highlights welcome messaging, study stats, continue-learning entry points, quiz and flashcard surfaces, and recent notebook activity. 
Sidebar navigation is organized around Dashboard, Library, Quizzes, Flashcards, Playlists, recent activity, and authenticated account actions. 
Library 
The library is an organization workspace with category navigation, search, sorting, batch selection, notebook reassignment, and quick study entry points rather than a simple notebook gallery. 

Notebook Editor 
Top bar (EditorNavbar): "Back" button to dashboard, Editable Notebook Title, "Versions" history button, and "Review Mode" button.

Format toolbar: Fixed below the navbar, providing rich formatting tools including headings, lists, tables, links, highlights, alignment, task items, and extended document controls. 
Left sidebar (Outline): Collapsible sidebar containing the Header-based Outline for navigation. 
Center: Content area with resizable width (using drag handles). Renders the full notebook content in a focused editor. 
AI integration: Floating action button (AiFab) to toggle the Slide-in right panel (AiSidebar) for chat-based assistance, notebook rewrites, explanations, and study asset generation. 
Review Mode 
Full-screen, distraction-free layout. Rendered read-only content. Left sidebar with outline navigation, right-side read-only AI tools, and a bottom player bar for play/pause, playback speed, and listening progression. Text highlights remain in sync with device TTS playback. 

Quiz Interface 
Quiz pages present saved study sets with notebook linkage, best-score history, builder controls, attempt submission, and results review. 
Flashcards 
Flashcard pages provide deck management, notebook-linked card sets, study mode, and mastery tracking across attempts. 
Playlists 
Playlist screens manage named listening queues of notebooks, including add/remove controls, ordering, and current-item continuation. 
Profile & Settings 
Authenticated settings support profile edits, password change, and AI configuration management. 

7.2 Mobile Application Wireframes









Dashboard (Home Screen)
Mobile dashboard is the continuation hub for recent study activity. It prioritizes returning to recent notebooks, playlist sessions, and study tools from the authenticated account. 
Navigation can be presented through a drawer or other mobile-friendly shell while preserving access to Dashboard, Library, Playlists, Quizzes, Flashcards, and Profile. 
Library
Mobile library supports category browsing, notebook search, and quick actions for opening, organizing, and sending notebooks into study flows. 
Notebook Editor
The mobile blueprint keeps notebook viewing, title management, review entry, and lightweight editing support aligned with the notebook-centered web experience. 
Outline access and AI assistance are expected to remain notebook-scoped even

when the mobile surface exposes a simplified interaction model. 
Review Mode 
Review Mode on mobile is optimized for continuation and listening: read-only notebook rendering, simple playback controls, preserved position, and quick jumps back into notebook sections. 
Study Playlists 
Mobile playlists continue the same named queue model used on web, allowing users to resume a listening order and move between notebooks as a planned study sequence. 
Quizzes & Flashcards 
Quiz and flashcard access on mobile follows the same notebook-linked study model, with concise interaction patterns for attempts, scoring, and continued review. 
Profile Screen 
Profile access includes account information, sign-out, and settings continuity for password and AI-related preferences as the mobile surface expands.
